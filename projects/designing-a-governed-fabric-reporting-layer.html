Designing a Governed Fabric Reporting Layer

I worked on moving ADA reporting from direct SharePoint to Power BI reporting into a more governed Microsoft Fabric structure. The work focused on centralising reporting data in a Lakehouse, using Python notebooks for Bronze, Silver and Gold transformation layers and connecting Power BI to cleaner reporting ready tables.

SharePoint source files Python notebooks Bronze, Silver and Gold layers Power BI curated outputs

Architecture

Business files stayed in SharePoint. Reporting logic moved into Fabric.

ADA reporting data lived across SharePoint folders and spreadsheet based team files. Before Fabric, Power BI reports were linked closer to source files, which made consistency, transformation control, refresh reliability and governance harder to scale.

The Problem

The reporting estate was growing faster than the process behind it.

Executive summary, health and safety, planning, design, OFCI, commercial, schedule, quality, environmental and commissioning data had separate source files and reporting requirements. Each area could exist, but the process was harder to audit and harder to repeat.

What I Built

A medallion reporting layer inside Microsoft Fabric.

Source data was brought into a Lakehouse and processed through controlled transformation stages. Bronze preserved raw tables, Silver applied cleaning and standardisation, and Gold produced reporting ready tables for Power BI.

01

Data source

SharePoint folders and discipline files such as executive summary, health and safety, planning, design, OFCI, commercial, schedule, quality, environmental and commissioning.

02

Fabric transformation

Python notebooks and pipelines moved data through controlled transformation stages instead of keeping business logic scattered inside individual Power BI files.

03

Bronze layer

Raw source tables landed with minimal transformation. This preserved the original data and gave the team an auditable starting point.

04

Silver layer

Data was cleaned, standardised and prepared. This included aligning naming, shaping columns and applying business rules needed before reporting.

05

Gold layer

Final reporting ready tables were created for Power BI. These tables were cleaner, more consistent and easier to reuse across reporting pages.

06

Security and governance

Azure Active Directory groups and governance registers documented access, ownership, refresh logic and report status.

Fabric Transformation

Python notebooks held the repeatable business logic.

The transformation layer separated extraction, cleaning and reporting outputs, which made the process easier to debug and explain.

Governance Evidence

The layer was documented, not just built.

The governance workbook tracked Fabric artefacts, table mappings, report status, workspace ownership and security groups. That made the build easier to explain and easier to hand over.

Access Control

Reports and audiences were mapped clearly.

Report status and security groups were documented so access did not depend on informal instructions.

What It Changed

Transformation moved out of scattered report files and into a controlled Fabric layer.

Reports became connected to documented tables. Lineage became easier to trace. Refresh logic became easier to manage. Access became mapped through defined security groups.
