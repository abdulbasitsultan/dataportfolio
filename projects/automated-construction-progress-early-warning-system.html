Automated Construction Progress Early Warning System

I built a Python based forecasting pipeline that moved construction progress reporting from manual baseline versus update comparisons into a reproducible early warning workflow. The system transformed planned and actual cumulative progress data, engineered time aware features and forecasted next step variance at package level.

R² 0.994 on weekly test data MAE 0.012 5 feature families engineered Ridge beat all Random Forest variants

The Problem

Progress reporting was descriptive and manual.

Planned and actual progress existed, but it showed what had already happened. I reframed the task as predicting next step variance so teams could see where the next update was likely to land.

What I Built

A reproducible forecasting pipeline.

I reshaped wide progress sheets into long format, created variance as actual progress minus planned progress, engineered current state, lag, speed gap, rolling trend and time order features, then compared Ridge Regression with Random Forest using chronological splits.

Data Movement

From workbook to model ready data.

The original file was designed for people, not forecasting. Dates sat across columns and planned and actual values lived in separate sheets. The pipeline made time a proper variable.

Signal

The planned and actual curves were pulling apart.

The widening gap made the forecasting task meaningful. Lagged variance became the strongest signal because packages drifting from plan tended to carry that behaviour forward.

Model Comparison

Ridge was simpler and stronger for this dataset.

Random Forest was tested as a non linear comparison, including deeper configurations. The best Random Forest reached R² of 0.970, while weekly Ridge reached 0.994 with lower error. The result showed that interpretable models can be stronger when the data structure is stable and explainability matters.

Output

Forecasts landed where project teams manage the work.

The pipeline produced package level predictions, not only aggregate summaries. That makes the early warning useful because someone can see which package needs attention.

What It Changed

The project moved reporting from hindsight to early warning.

The work showed how planned and actual progress data already available in a project environment can become a forecasting signal when the data is reshaped, features are engineered properly and model choice is driven by the structure of the problem.
